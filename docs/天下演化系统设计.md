# 残明余烬 1.7：天下演化系统设计

## 1. 系统定位

天下演化是主模型背后的世界状态机与旁线叙事台。

- 主模型只负责玩家当前视角内的正文、人物表演与 MVU 更新。
- 天下演化副模型在每轮正文结束后进行一次请求，只提交本轮发生的世界变化和两段旁线场景。
- 脚本保存完整世界档案、校验增量操作、生成下一轮联动包，并在天下演化面板中展示平行世界。
- 平行世界原文不得写回主模型聊天楼层，避免下一轮主模型模仿 `<平行世界>` 标签或旁线文风。

玩家输入只代表行动意图。只有主模型正文、最终 MVU 状态和已经通过天下演化校验的远方变化，才能成为客观事实。

## 2. 设计原则

### 2.1 单次调用

一轮天下演化只调用一次副模型。请求同时返回：

1. 本轮世界状态增量操作；
2. 最多两个可展示的平行世界场景。

不为了核账、推进和写作分别调用多个模型，避免额外等待。

### 2.2 模型只写变化

完整世界档案始终由脚本保存。副模型不得每轮重写整个档案，只返回本轮必要变化。

没有变化的事件、人物、情报、伏线和摘要不得重复输出。

### 2.3 使用稳定 ID，不使用数组下标

事件、人物、驿报、伏线通过稳定 ID 更新。禁止使用 `/activeEvents/3/stage` 一类依赖数组位置的通用 JSON Patch。

例如：

```json
{
  "type": "event.patch",
  "id": "EVENT-漕粮亏空",
  "set": {
    "stage": "县衙立案",
    "next_trigger": "捕役开始搜查涉案粮铺"
  }
}
```

### 2.4 脚本掌握最终写入权

副模型只能提出操作。脚本负责：

- 校验操作类型和必填字段；
- 检查目标 ID；
- 合并已有记录；
- 拒绝不完整或越权的操作；
- 按容量限制归档；
- 生成镜头历史；
- 生成下一轮联动包；
- 一次性写入完整新状态。

### 2.5 分区容错

单条非法操作只影响该操作，不使整轮作废。

- 合法操作继续应用；
- 非法操作跳过并记录原因；
- 平行世界缺失不影响合法档案变化；
- 档案中个别操作无效也不影响合法平行世界；
- 顶层结果完全无法解析时才重试整次请求。

## 3. 每轮闭环

1. 主模型收到上一轮压缩后的天下演化联动包。
2. 主模型生成玩家视角正文与 MVU 更新，不生成平行世界。
3. 脚本记录 `chatId + messageId + swipeId + messageHash + baseRevision`。
4. 正常情况下，脚本在 MVU `VARIABLE_UPDATE_ENDED` 后开始结算；若未收到事件，使用兜底定时器。
5. 脚本构造本轮请求：
   - 玩家输入，仅作为意图；
   - 当前主模型正文；
   - 最终 MVU 时间、地点与本轮差异；
   - 最近数轮只读上下文；
   - 与当前人物、地点、势力相关的工作集；
   - 世界摘要与少量近期事实。
6. 副模型一次返回 `operations` 和 `parallel_scenes`。
7. 脚本逐条验证操作，在旧状态副本上应用合法变化。
8. 脚本从合法变化生成 `nextTurnPacket`，从旁线场景生成 `cameraHistory`。
9. 脚本再次校验聊天、楼层、重抽页、正文哈希和档案 revision。
10. 校验通过后，一次性保存完整新状态。
11. 平行世界仅进入天下演化档案和 UI，不修改主模型消息。
12. 脚本刷新深度 0 的下一轮主模型联动提示。

首楼 `messageId = 0` 是开场和初始化内容，不自动推演；只有用户主动点击“重新推演”时才允许处理。

天下演化在后台运行，不得在 `GENERATION_AFTER_COMMANDS` 中长时间阻塞下一轮正文。玩家在结算完成前立即继续时，可以暂时使用上一版联动包；新结果完成后供之后轮次使用。

## 4. 副模型输出协议

```json
{
  "schema_version": 2,
  "base_revision": 17,
  "operations": [
    {
      "type": "fact.add",
      "id": "FACT-桐城-0017",
      "value": {
        "content": "县衙正式发出缉捕文书",
        "status": "occurred",
        "scope": "player_scene",
        "location": "桐城县",
        "actors": ["知县"],
        "witnesses": ["县衙书吏"],
        "publicity": "县衙内部公开",
        "confidence": 0.92,
        "importance": 70,
        "evidence": "主模型正文明确写出文书已经发出"
      }
    },
    {
      "type": "event.patch",
      "id": "EVENT-漕粮亏空",
      "set": {
        "stage": "县衙立案",
        "summary": "缉捕文书已经下发",
        "next_trigger": "捕役开始搜查涉案粮铺"
      }
    }
  ],
  "parallel_scenes": [
    {
      "location": "桐城县·林记米铺后院",
      "time": "酉时",
      "actors": ["周氏"],
      "action": "收起与涉案粮铺有关的旧账",
      "body": "……"
    }
  ]
}
```

`base_revision` 必须等于请求中的档案 revision。脚本不信任模型回传的聊天 ID 或楼层号，而使用请求任务本身绑定的信息。

## 5. 增量操作

### 5.1 事实

#### `fact.add`

新增本轮已经发生的客观事实。

必须满足：

- `status` 为 `occurred`；
- `confidence >= 0.6`；
- 有非空 `content` 与 `evidence`；
- 不得把计划、命令、传闻、失败尝试和纯气氛写成事实。

### 5.2 事件

- `event.upsert`：创建新的持续事件，必须提供标题、阶段、地点、摘要和下一触发条件。
- `event.patch`：只提交已有事件发生变化的字段。
- `event.resolve`：结束事件并从活跃事件中移除。

`event.patch` 的 `id` 不存在时拒绝，不自动创建空事件。

### 5.3 人物

- `actor.upsert`：创建确有当前行动的重要人物。
- `actor.patch`：更新已有角色的位置、目标、当前行动、知识、下一决策或更新原因。

不得仅因人物出现在角色资料或名册中就建档。

### 5.4 情报

- `intel.upsert`：创建在途信息包。
- `intel.patch`：更新传播位置、状态、预计抵达时间和知情者。
- `intel.remove`：情报失效、被截获或已经完成传播后移除。

静态秘密和人物心意不属于在途情报。

### 5.5 伏线

- `hook.upsert`：创建有触发条件和失效条件的伏线。
- `hook.patch`：推进阶段、征兆或条件。
- `hook.resolve`：伏线已经触发、失效或不再需要追踪。

### 5.6 世界摘要

`summary.replace` 只在以下情况使用：

- 重大事件成立或结束；
- 世界日期明显推进；
- 原摘要已不再准确；
- 多条旧事实需要压缩归档。

普通回合不得重写世界摘要。

## 6. 脚本派生字段

以下字段不由副模型重复输出。

### 6.1 `clock`

以本轮最终 MVU `世界运转` 字段为准。

### 6.2 `cameraHistory`

从 `parallel_scenes` 的地点、人物和行动生成，保留最近若干条，用于防止连续重复镜头。

### 6.3 `nextTurnPacket`

由脚本根据本轮合法操作生成：

- `hardFacts`：本轮新增事实；
- `arrivedIntel`：本轮抵达或公开的情报；
- `localConsequences`：当前地点附近的事件变化；
- `npcKnowledge`：本轮人物知识变化；
- `activePressures`：本轮推进的活跃事件；
- `cameraCandidates`：旁线场景留下的后续候选；
- `constraints`：固定知识边界与本轮不得提前泄露的内容。

下一轮主模型不会收到平行世界原文。

## 7. 平行世界

### 7.1 内容要求

- 每轮最多两个玩家视野之外的场景；
- 每段包含地点、时点、人物、行动和正文；
- 优先展示本轮实际推进的事件、人物行动或情报传播；
- 素材不足时可以写低风险日常切片；
- 不得重演玩家当前场景；
- 不得凭空制造重大胜负、死亡、陷城和政局结果；
- 不使用 `<平行世界>` 标签和元叙事转场。

### 7.2 保存与展示

平行世界按回合保存在聊天级天下档案中，记录：

- 来源楼层和重抽页；
- 生成时间；
- 档案 revision；
- 最多两个场景；
- 本轮接受与拒绝的操作数量。

面板提供独立“旁线”栏目。重新推演同一楼层时替换该楼层原有旁线，不重复追加。

旧聊天楼层中已经存在的 `<平行世界>` 保持原样，不主动修改；新版本不再写入新的标签块。

## 8. 工作集与长期记忆

每轮不向副模型发送全部档案，而发送相关工作集：

- 世界摘要；
- 当前正文命中的活跃事件；
- 当前涉及或正在行动的重要人物；
- 临近抵达或命中相关地区的情报；
- 可能被本轮条件触发的伏线；
- 最近事实和关键词召回事实；
- 最近镜头历史。

工作集只影响请求体大小，不删除完整档案。完整档案仍保存在聊天变量中。

默认上限：

- 活跃事件 24 条；
- 重要人物 48 名；
- 在途情报 60 条；
- 活跃伏线 16 条；
- 近期确认事实 240 条；
- 镜头历史 18 条；
- 平行世界回合 24 轮；
- 检查点 8 个。

## 9. 一致性与并发

每个任务绑定：

```text
chatId + messageId + swipeId + messageHash + baseRevision
```

写入前再次比较。任一字段变化，返回结果视为过期并丢弃。

- 同一时刻只执行一个天下演化请求；
- 新任务到来时只保留最新待处理任务；
- 切换聊天时停止旧请求、清空定时器和待处理任务；
- 重抽或编辑从目标楼层之前的检查点恢复；
- 删除楼层后恢复到最近有效检查点；
- 保存前先在内存副本中应用全部合法操作，不边解析边写变量。

## 10. 请求与失败策略

- 默认温度：`1`；
- 默认最大输出：`10000`；
- 独立 API 与当前酒馆连接均应尽可能使用同一组模型参数；
- 请求超时后主动停止对应 generation ID；
- 顶层 JSON 或结构完全无法解析时最多自动重试一次；
- 用户主动停止后不重试；
- 单条操作失败不重试整个请求；
- UI 显示错误分类、最后成功楼层、接受操作数和拒绝操作数。

错误分为：

1. 连接或请求错误；
2. 顶层解析错误；
3. 操作校验错误；
4. 过期结果；
5. 保存错误。

## 11. 设置迁移

设置版本由 2 升级为 3。

- 未保存温度或仍使用旧默认 `0.45` 的用户迁移到 `1`；
- 未保存最大输出或仍使用旧默认 `5200` 的用户迁移到 `10000`；
- 用户明确保存的其他数值保持不变；
- 已有聊天档案在读取时补充空的平行世界历史，不要求手工清档。

## 12. 界面

视觉方向继续使用“明末值房案牍”：

- 宋体气质只用于品牌标题和少量印章；
- 长正文、设置、事件和档案使用清晰的无衬线中文字体；
- 总览、世事、旁线、档案和设置围绕真实数据组织；
- 旁线页面突出最新两段场景，并允许回看最近回合；
- 不在聊天正文中展示或折叠平行世界。

移动端要求：

- iframe 使用动态视口单位并考虑安全区；
- 顶栏只保留品牌和关闭按钮；
- 主内容单列滚动；
- 底部操作区固定为四个栏目加设置、重新推演；
- 设置抽屉在小屏变为全屏页面；
- 最终移动端媒体查询必须位于桌面覆盖之后，避免被后续规则反向覆盖；
- 不出现横向滚动；
- 触控按钮最小高度 40px。

## 13. 第一阶段实现范围

本轮实现：

- 单次副模型请求；
- ID 驱动的增量操作；
- 合法操作逐条应用；
- 脚本派生镜头历史和联动包；
- 平行世界保存到天下演化档案；
- 独立旁线 UI；
- 不再修改主模型消息；
- 设置版本 3 与新默认参数；
- 请求超时清理、待处理任务和错误统计；
- 移动端底栏、抽屉、滚动与安全区修复。

后续再评估：

- 天下演化与 MVU 并行请求；
- 严格故事时间调度器；
- 人物行程与资源约束；
- 别名和实体合并；
- 更强的历史语义检索。
