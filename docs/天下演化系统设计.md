# 残明余烬 1.7：天下演化系统设计

## 1. 目标

天下演化系统是主模型背后的“史官与调度台”。主模型仍然负责正文、NPC 表演与 `<平行世界>`
的文学表达；副模型在每次主回复结束后读取最新正文、最近数轮上下文和长期世界档案，提取已发生事实，推进远方世界，并为下一轮主模型准备一份精简的联动提示。

系统不根据玩家输入直接判定结果。玩家输入只代表意图，只有主模型正文明确写成已经发生的事情，才可能进入客观世界档案。

## 2. 每轮闭环

1. 主模型收到上一轮已经结算的“天下演化上下文”，生成本轮正文、MVU 更新和 `<平行世界>`。
2. 脚本等待主回复与 MVU 更新完成，读取：
   - 最新玩家输入；
   - 最新主模型完整回复；
   - 最近若干轮只读上下文；
   - 本轮 MVU 前后差异；
   - 上一世界检查点。
3. 副模型用结构化 JSON 返回：新事实、人物行动、事件、在途情报、伏笔变化，以及下一轮主模型联动包。
4. 脚本验证、去重并保存状态，随后更新深度 0 系统注入。
5. 下一轮重复上述流程。

主模型无需提交额外回执。副模型直接阅读正文，并区分“已经发生”“计划”“命令”“传闻”“假设”和“纯场景纹理”。

## 3. 记忆分层

### 3.1 最新一轮

完整读取，是本次新增事实的主要来源。

### 3.2 近期上下文

默认回看最近 3 个交互回合，用于消解“他”“那封信”“仍照昨日安排”等承接表达。近期上下文只用于理解，禁止再次提交旧事实。

### 3.3 长期世界档案

结构化保存：

- 世界总览；
- 活跃事件；
- 重要人物当前地点、目标、行动与知识；
- 情报传播队列；
- 活跃伏笔；
- 已确认事实；
- 最近平行世界镜头。

长期档案不在每轮全部发送。副模型主要读取活跃记录、近期事实，以及根据本轮人物、地点、势力和事件命中的旧记录。

### 3.4 原始聊天记录

正文和平行世界原文继续保留在酒馆聊天楼层。世界档案保存来源楼层与简短证据，需要时可回查原文，无需重复存储全文。

## 4. 信息边界

系统严格区分：

- **客观世界真相**：副模型维护，允许供 `<平行世界>` 使用；
- **主角认知账本**：现有 `天下地图`、`时局与任务.势力关系` 等 MVU 字段，只记录主角通过合理渠道获得的信息；
- **人物知识**：每名重要 NPC 只依据目击、对话、公文、书信、流言等渠道知情；
- **在途情报**：事件已经发生，但消息尚未抵达目标人物或地区。

人物对话只能证明“此人说过这句话”，不能自动证明话中内容属实。平行世界的叙述性事实默认属于客观历史，但其中人物说的话仍可能是误判、夸张或谎言。

## 5. 聊天与分支隔离

- API 和界面设置保存在本地脚本设置，可跨聊天共享。
- 世界档案只保存在 `{ type: 'chat' }` 的 `cmyj_world_engine_v1` 字段中。
- 每个结算任务绑定 `chatId + messageId + swipeId + messageHash`。
- API 返回时若聊天、楼层、重抽页或正文哈希已经变化，结果作废。
- 切换聊天时中止旧任务、清空运行时缓存、移除旧注入并载入新聊天档案。
- 重抽、编辑或删除旧楼层时，从较早检查点恢复并重新结算，不能把两个回复分支合并。

## 6. 时间推进

叙事回合不等于世界日期。系统使用三种触发：

- 每回合：人物日常、局部流言、镜头轮换；
- 剧情时间推进：行程、送信、伤病、军粮、工程等实际进度；
- 重大事件：公开处决、宣战、占城、开仓、发布政令等立即产生第一圈影响。

如果本轮剧情时间没有推进，平行世界可以展示同一时刻的其他切片，但不能让军队凭空走完数百里或让城池无因易手。

## 7. 历史时间线

时间线是默认惯性，不是强制剧本。后续版本应将重大历史节点编译成带有事件 ID、时间窗口、重要度、前置条件和偏离条件的模板。

第一版先把当前日期、世界总览、活跃事件和角色卡已有时间线作为副模型的约束；不允许副模型仅因历史上发生过某事，就跳过前置过程直接判定完成。

## 8. 数据规模与归档

第一版默认上限：

- 活跃事件 24 条；
- 重要人物 48 名；
- 在途情报 60 条；
- 活跃伏笔 16 条；
- 近期确认事实 240 条；
- 镜头历史 18 条；
- 最近检查点 8 个。

达到上限时优先清理已经解决、失效或重要度最低的记录。重要事实不得无声删除，应先压缩进世界总览或历史摘要。

## 9. 主模型联动包

下一轮注入只包含主模型真正需要的内容：

- 当前客观硬事实；
- 本轮可能抵达的情报；
- 当前地点可观察到的后果；
- 相关 NPC 的知识边界；
- 正在施压的活跃事件；
- 平行世界候选镜头；
- 禁止提前确定或泄露的内容。

注入启用 `should_scan`，使其中的人名、地名和势力名可以激活对应世界书条目。

## 10. 副模型连接方式

系统支持两种连接方式，二选一即可：

- **使用当前酒馆**（默认）：不读取 URL 和密钥，也不把密钥复制到脚本。`generateRaw`
  不指定连接字段，由酒馆使用当前接口与当前模型完成请求；界面只读取并显示酒馆安全公开的接口来源和模型名称。
- **使用独立 API**：填写 API 地址、密钥和接口类型后，通过“获取模型”按钮调用酒馆助手的
  `getModelList`；若该接口不可用，再尝试标准 `/v1/models`。返回的模型放入下拉列表，同时保留手动输入模型名的能力。

结构化推演所需的系统提示、用户提示和 JSON
Schema 始终由天下演化脚本提供。即使使用当前酒馆连接，也不会调用正文写作预设；复用的只是连接与模型。需要让副模型和主模型使用不同服务时，再切换到独立 API。

## 11. 界面结构

- 参考图只用于提取信息结构，不做逐像素复刻：保留横向工作台比例、主次栏关系和连续事件流；不照搬时辰轴、示例文案、装饰位置或与本系统无关的控件。
- 天下演化自身的信息优先级高于参考图：连接状态、结算状态、伏线、延后后果、重建联动和设置入口必须围绕实际数据与工作流重新组织。
- 界面直接读取状态栏的 `canming-afterglow-statusbar:theme`，同步日、夜、星、墨四套主题；圆角、卡片、状态点、胶囊标签、按钮层级和阴影尺度沿用状态栏设计语言。
- 入口继续采用可拖动的罗盘悬浮按钮，配色跟随状态栏主题，图形与状态栏入口明确区分。
- 主面板采用单页工作台：顶部显示标题、当前日期、连接与值房状态以及叉号关闭入口；桌面窗口限制为较紧凑的 1480 × 790 上限。
- 总览首屏先展示“今日天下”摘要和四项状态数量，不设置十二时辰轴或其他无实际操作价值的装饰时间线。
- 活跃事件使用连续时间轴列表，右侧只保留“正在展开的伏线”和“可能延后的后果”两组批注；减少卡片嵌套。
- 总览、世事、档案、设置和立即推演集中在底部操作坞；状态回执与重建联动置于次级行，“立即推演”是唯一强调的主操作。
- 设置不再占用主内容页，改为右侧抽屉；抽屉内仍提供当前酒馆连接、独立 API、模型获取、自动结算和档案管理。
- 正文与设置全部使用常规可读中文字体。标题可使用宋体风格，但不得让篆隶等装饰字体承担信息阅读。
- 装饰图片不参与主内容渲染，避免为了贴近参考图制造割裂感。宽屏保持事件流与右侧批注并列，窄屏自动改为单列且不产生横向滚动。

## 12. 失败原则

- 副模型调用失败不得影响主回复和 MVU。
- 非法 JSON 最多自动重试一次。
- 写入采用完整新状态替换，避免留下半份结果。
- 低可信度提取不写入硬事实，可暂存为传闻或待确认事件。
- 界面始终显示最后一次成功结算、当前任务和错误信息。

## 13. 第一版范围

第一版实现：

- 独立聊天档案；
- 最近 3 轮上下文读取；
- 结构化副模型推演；
- 事实、事件、人物、情报、伏笔和镜头记录；
- 下一轮主模型注入；
- 自动结算与手动推演；
- 重复结算保护；
- 聊天切换和异步回包隔离；
- 罗盘悬浮入口与管理面板；
- 自定义 API 模型列表获取与可选酒馆代理连接；
- 世界档案导出、当前聊天档案清空。

后续版本再完善历史事件模板编译、跨楼层分支重演、语义检索和历史档案压缩。
